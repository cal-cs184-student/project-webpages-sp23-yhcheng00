Project 2

Overview

In this project, I designed a Bezier curve evaluator using de Casteljau’s algorithm to interpolate between the control points for a parameter t until it converged to a point on the curve. This was then used by the render to evaluate and display the curve and update it in response to movement of control points on the GUI. Next, the concept of Bezier curve was extended to Bezier surfaces, by chain-evaluating multiple Bezier curves for parameter u and using the result at each curve as control points to generate and evaluate a final curve for parameter v. In this fashion, a surface evaluated at u and v could be created. I further implemented area-weighted vertex normals, giving each vertex an attributed normal vector that is an average of the adjacent faces, weighted by the respective area of the faces. This allows shading that linearly interpolates between vertices and is much more even than simply shading each face. Implementing edge flips and edge splits required understanding the half-edge structure of a mesh, in which half-edges form the main data structure that links elements of the mesh together. Edge flips and splits are highly generalizable, making the algorithm straightforward, but careful reassignment of values for vertices and edges is needed to ensure success. Both operations form the basis of loop subdivision, a process of refining a mesh procedurally by creating new faces in a 4:1 fashion (splitting each face into 4 by splitting each edge and creating new edges). Updated old vertex positions are calculated with a weighted average of neighboring vertex positions and current positions, while new vertex positions are calculated based on neighboring vertex positions.

This project helped me to truly understand and visualize Bezier curves, while I only had a cursory understanding before. Furthermore, I understand how loop subdivision can be used to approach shapes, though its tendency to round sharp features means it is difficult to approximate certain shapes.

Part 1

de Casteljau’s algorithm is a recursive method to evaluate Bezier curves by multiple linear interpolation: Since Bezier curves can be constructed by multiple linear interpolation of user-defined control points with a parameter t, de Casteljau’s algorithm interpolates a curve of degree n with n+1 control points to arrive at n points evaluated at parameter t along each component line. It treats this as another Bezier curve and evaluates similarly, performing this recursively until it arrives at one point, which is defined to lie along the Bezier curve at parameter t.

Since the starter code implements the application of de Casteljau’s algorithm recursively, I simply had to fill in the recursive step of returning n control points from n + 1 control points. I did this by iterating through every point (stopping short of the last point, as I needed to interpolate the current point with the one ahead of it, impossible for the last point) and interpolating between the ith position and i+1 position for the provided parameter t. This position would be inserted to the end of a 2D vector which was returned at the end of the loop.

Part 2

Instead of a single parameter t, the surface takes in two parameters, u and v. Rather than having n control points along 1 dimension like a Bezier curve, a Bezier surface has n by n control points. Treating this as a n by n matrix, each column therefore has n values, each forming a control point.

Therefore, each column has a Bezier curve of those n control points. Evaluating each curve at u with de Casteljau’s algorithm results in n points that form a Bezier curve representing the surface for a given u and all values of v. To evaluate this surface at a specific v, apply de Casteljau’s algorithm again on this curve, but with parameter v to arrive at a point on the surface evaluated for u and v.

While de Casteljau’s algorithm is recursive by nature, I chose to apply the recursive step iteratively, with evaluate1D() reusing a variable to hold a vector of 3D positions and continuing to apply the recursive step developed in Part 1 until I arrived at a single point for a column. In evaluate(), I repeated this for all columns to get a curve for a given u and storing it in a vector of 3D vectors, and then applied evaluate1D() to this curve with v as a parameter, yielding a point on the surface for u and v.

Part 3

To implement area-weighted vertex normals, I had to cycle through every single face bordering the vertex. In order to do this, I took advantage of the fact that every outgoing half-edge has a unique face associated with it (i.e. no half-edges sharing the same vertices will share a face for this application). Therefore, by iterating through outgoing half-edges, I could iterate through each face. This could be done as exemplified in the half-edge primer, by simply assigning the current half-edge h to h→twin()→next() each cycle until arriving back at the same half-edge. In between assignment, I took the cross product of two of the half edges and divided by half to get the face’s area, and got the normal vector of the face using the Face’s built-in normal() method, adding this to the total normal vector. I then returned the unit version of this total normal vector accumulated from the area-weighted face normals. Part 4

To implement the edge flip operation, I took the half-edge of the edge to be flipped to be the “left” half-edge, soon to be the top half-edge, while making its twin the “right” half-edge soon to be the bottom. I traversed the quadrilateral according to the old hierarchy before flipping in order to find the relevant vertices (a, b, c, d) and half-edges (ca, ab, bd, dc). I then set every single attribute of each object (the aforementioned vertices, half-edges, and left and right edges) to the new configuration after flipping as shown in the project guidelines. Although many of these assignments were likely redundant, I wanted to avoid bugs that would arise from failing to reassign something that was necessary. After reassignment, the easiest way to guarantee the faces had a consistent half-edge was to tie them to the left and right edges. I thus retrieved the leftFace and rightFace from the left and right edges before flipping, and assigned their half-edge values to leftEdge and rightEdge respectively. This way, the faces were guaranteed to be associated with the correct edge, and vice versa.

I initially attempted to implement this by only assigning what I felt was necessary to perform the flip, but the result refused to render. After an hour of debugging (thinking of various things I may have forgotten) I started over with total reassignment, and it worked.

Part 5

Edge splits are initially conducted similarly to edge flips. As with the edge flipping method in Part 4, I traverse to find the same vertices and half-edges. However, I create a new vertex m as the midpoint between b and c, and create new edges (am, md, bm, ma, dm, mb). The leftEdge and rightEdge are reassigned to mc an cm. I then create three new edges: south, east, and west, with the original edge assigned to north. The left and right faces are assigned to northeast and northwest, and two new southwest and southeast faces are created. I then reassign in a similar fashion as Part 4, but this time with the additional created vertex, edges, and faces, following the diagram from the project guidelines.

As I implemented a total reassignment strategy here, similar to before, I encountered no bugs of note.

Part 6

My implementation of loop subdivision mirrored the numbered checklist provided by the starter code almost exactly.

I first computed new vertex positions and stored them in Vertex::newPosition. I calculated the degree of a vertex and neighboring position by iterating through every neighboring vertex using a similar method to printVertexPositions in the Half Edge primer, using h_twin→vertex() as the neighboring vertex. However, I didn’t bother marking them as part of the original mesh, since that distinction would be made in the isNew attribute when an edge is split.

Each edge also received a calculated position for its vertex in newPosition, using multiple iterations of next() on itself as well as its twin() to gather the relevant vertices and their positions.

I then iterated through edges to split them, and set the vertex returned to the edge’s newPosition. However, I would only split if there was a valid noninfinity newPosition x value (The splitEdge function was modified such that new edges would be created with INFINITY for x, y, and z), preventing an infinite loop.

This also allowed me to set the isNew variable in the splitEdge function and use it to determine which edges to flip, instead of to determine which edges were not to be split.

Finally, I iterate through the vertices and copy the new positions into the position attribute.

Above is a loop subdivision of the provided icosahedron. Notice that loop subdivision tends to smooth and round out shapes, since each vertex receives a new position based on an (weighted) average of its neighbors.

This has a profound effect on sharp corners and edges, as they tend to regress towards the mean of the positions around them. It can be reduced by pre-splitting edges connected to the vertex forming the sharp corner. This creates new neighboring vertices in the mesh that are closer to the corner, reducing its change in position upon update.
